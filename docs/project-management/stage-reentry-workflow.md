# LandPPT 阶段可重开（Stage Reentry）项目管理方案

## 1. 目标与适用范围
- 目标：在 LandPPT 项目中建立“阶段完成后可重开”的标准流程，避免因需求变更、质量回归、环境差异导致的返工失控。
- 适用范围：`/app/LandPPT` 仓库内的需求、开发、测试、发布等阶段性工作；适用于功能开发、缺陷修复、重构和配置变更。
- 不适用：紧急线上止血（先修复后补流程），但止血完成后必须补登记并纳入重开统计。

## 2. 阶段状态机（含 Reopen）

### 2.1 状态定义
- `NotStarted`：未开始。
- `InProgress`：执行中。
- `ReadyForAcceptance`：已提交，待验收。
- `Completed`：验收通过，阶段关闭。
- `Reopen`：阶段重开，进入复核与重执行流程。

### 2.2 状态流转
`NotStarted -> InProgress -> ReadyForAcceptance -> Completed`

`Completed --(满足重开条件)--> Reopen -> InProgress -> ReadyForAcceptance -> Completed`

约束：
- `Reopen` 只能从 `Completed` 进入。
- 每次重开必须关联一次“重开记录”（原因、范围、责任人、计划时间）。
- 未完成重开必做清单，不得从 `Reopen` 直接回 `Completed`。

## 3. 触发重开条件
出现以下任一情况即可触发：
1. 需求变更：产品目标、验收口径、关键业务规则发生变化。
2. 缺陷回流：已完成阶段发现 P1/P2 缺陷，或同类缺陷重复出现。
3. 依赖变化：上游接口、第三方服务、配置策略变更影响当前阶段结论。
4. 验收证据不足：验收记录缺失、样本不足、结果不可复现。
5. 线上反馈：灰度/生产出现与阶段结论矛盾的问题。

触发门槛建议：
- 任一 P1 缺陷：立即重开。
- 24 小时内同模块新增 2 个及以上 P2：重开评审。
- 验收无法复现：默认重开。

## 4. 重开后必做清单
重开后按顺序执行，缺一不可：
1. 信息复核
   - 复核需求版本、设计稿、接口契约、配置项。
   - 确认“当前目标”与“首次完成时目标”的差异。
2. 差异分析
   - 输出差异清单：代码、配置、数据、依赖、环境。
   - 标注影响范围：模块、接口、用例、发布风险。
3. 关键步骤重跑
   - 按阶段关键路径重跑（构建、核心流程、回归用例、必要脚本）。
   - 对历史失败点做定向复测。
4. 重新验收
   - 按最新验收标准执行。
   - 更新验收证据：日志、截图、结果链接、测试报告。

## 5. 进入/退出标准

### 5.1 进入 Reopen 标准
- 已处于 `Completed`。
- 存在明确触发条件并被 PM 记录。
- 已定义重开范围（全量/局部）和责任人。

### 5.2 退出 Reopen（回到 Completed）标准
- 重开必做清单全部完成。
- 差异项已关闭或有书面风险接受结论。
- 回归结果达标，无阻断缺陷。
- PM、研发、测试三方确认并留痕。

## 6. 角色职责（PM / 研发 / 测试）

### PM
- 判定是否重开，控制范围和优先级。
- 维护重开记录与节奏（开始/截止时间）。
- 组织重新验收并给出关闭结论。

### 研发
- 提交差异分析与修复方案。
- 执行关键步骤重跑，补充技术证据。
- 修复后同步变更说明与影响面。

### 测试
- 基于差异清单更新测试范围。
- 执行回归与重点冒烟，输出结论。
- 统计回归缺陷并反馈重开原因归类。

## 7. 指标与复盘口径
按迭代/月份统计：
1. 重开次数
   - 定义：`Reopen` 进入次数。
2. 重开原因分布
   - 口径：需求变更/缺陷回流/依赖变化/证据不足/线上反馈。
3. 回归缺陷率
   - 公式：`回归阶段发现缺陷数 / 重开后执行用例总数`。

建议阈值（可按团队成熟度调整）：
- 单需求重开次数 > 2：必须复盘。
- 缺陷回流类重开占比 > 40%：强化开发自测与准入检查。
- 回归缺陷率连续两期上升：冻结新增需求，先治理质量。
